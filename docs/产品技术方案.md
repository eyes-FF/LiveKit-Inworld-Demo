> 版本:v1.0(2026-06-11) · 状态:已上线

## 1. 产品定义

Web 端实时语音 AI 助理。用户在浏览器点一个按钮即可与 AI 进行自然语音对话,支持打断。验证两项核心能力:

1. **Inworld 语音链路**:Inworld STT(识别)+ Inworld TTS(合成)构成的 pipeline 式语音对话
2. **实时上下文注入(RAG)**:在 LLM 生成回答之前,根据用户刚说完的话,毫秒级检索并注入知识片段与风格示例,且注入过程在界面上完全可观测

### 1.1 用户可用的全部功能

| 功能 | 说明 |
|---|---|
| 实时语音通话 | 点「开始通话」即接通,WebRTC 双向音频,支持自然打断(用户开口 AI 即停) |
| 双向字幕 | 用户与 AI 的话实时转成文字气泡,每条带出现/结束时间戳(HH:MM:SS.mmm) |
| 响应延迟徽章 | 每轮 AI 回复第一段上显示「用户说完 → AI 开口」毫秒数,绿(<1.5s)/琥珀(<3s)/红(≥3s)三级配色 |
| 注入记录窗口 | 逐条显示本通电话每次注入的具体内容(知识/示例 chip + 原文 + 相似度分数 + 毫秒时间戳) |
| 注入量统计 | 右栏实时显示最近一轮注入的条数与字符数(≈tokens) |
| 对话语言 | 中文 / English,联动 STT 语言、音色列表、AI 指令与开场白语言 |
| 音色选择 | 下拉选 Inworld 内置音色(中文 10 个/英文 121 个,带描述);或自定义:填 voice ID 直接用,填英文描述则自动生成专属音色(~15s) |
| 语速 / 随机性 | TTS speaking_rate(0.5–2.0×)与 temperature(0–2) |
| 人设(persona) | 粘贴任意 system prompt(≤10000 字)定义 AI 是谁、怎么说话;开场白按人设即兴生成 |
| few-shot 注入开关 | 现场开/关对照注入效果 |

设置面板为 Figma 式右侧属性栏:可整体收起/展开;通话中全部锁定,结束通话后可修改,下次通话生效。

### 1.2 非目标

用户系统、权限、生产级持久化、横向扩缩容、自托管 LiveKit Server。

## 2. 总体架构

```
┌─────────────────────────────────────────────────────────────────┐
│ 浏览器前端  Next.js 16 + @livekit/components-react 2.9           │
│  通话 UI · 字幕/延迟/注入观测 · 设置面板                          │
└──────┬───────────────────────────────┬──────────────────────────┘
       │ ① POST /api/token             │ ② WebRTC 音频(双向)
       │   (携带全部会话设置)            │
       ▼                               ▼
┌─────────────┐                ┌──────────────────┐
│  Demo API    │                │  LiveKit Cloud    │
│  FastAPI     │                │  (托管,不自建)     │
└──────┬───────┘                └────────┬─────────┘
       │ ③ /api/session-config           │ ④ Job 派发 + 音频流
       │   /api/context (HTTP,同容器)     ▼
       │                        ┌──────────────────────────────────┐
       └───────────────────────▶│  Agent Worker  livekit-agents 1.5 │
                                │  inworld.STT → openai.LLM →       │
                                │  inworld.TTS                      │
                                │  on_user_turn_completed ←★注入点  │
                                │  publish_data ──→ 前端观测数据     │
                                └──────────────────────────────────┘
```

部署形态:**前端 Vercel + (API + Agent)单 Docker 容器 Railway**。API 与 Agent 同容器共生,检索调用走 localhost,保住本地检索的毫秒级延迟。

### 2.1 关键架构决策

| 决策 | 选择 | 原因 |
|---|---|---|
| S2S 实现 | 拆分 pipeline(STT→LLM→TTS),不用原生多模态 RealtimeModel | 注入钩子 `on_user_turn_completed` 只在 pipeline 模式存在;audio→audio 模型中间没有可拦截的文本轮次 |
| LLM 编排位置 | LiveKit 侧自己拼 `inworld.STT()`/`inworld.TTS()` 插件,不用 Inworld Realtime WebSocket 黑盒 | 黑盒模式 LLM 在 Inworld 侧编排,注入钩子够不着 |
| 检索引擎 | 本地 embedding(fastembed/ONNX)+ 内存 numpy 向量索引,不上外部向量库 | 注入同步阻塞在 LLM 生成之前,检索必须 <50ms;实测本地 ~3.5ms,外部向量库一次网络往返就破预算 |
| 长文本传输 | 会话配置存 API 侧(room → 配置表),agent 按房间名取;token 不带 metadata | 中文经 JSON 转义 6 字节/字,10K 字 ≈ 60KB 会顶爆 LiveKit 64KB metadata 上限,且 token 走 URL 传输 |
| 打断 | 框架内置(`allow_interruptions=True` + 参数调校) | LiveKit 1.5+ 自带 Adaptive Interruption Handling;`False` 有已知 bug |

## 3. 核心流程

### 3.1 通话建立

```
用户配置设置面板 → 点「开始通话」
→ 前端 POST /api/token {room, identity, voice, persona, rate, temp, lang, shots}
→ API:
   a. voice 是描述文字? → 调 Inworld Voice Design(design + publish)生成专属 voiceId(~15s)
   b. 全部设置存入 SESSION_CONFIGS[room](内存,LRU 200 条)
   c. 签发 LiveKit AccessToken(仅 room_join 授权,312 字节)
→ 前端 <LiveKitRoom> 连 LiveKit Cloud
→ LiveKit 派发 Job 给 Agent Worker
→ Agent: GET /api/session-config?room=… 取配置
   → 构造 inworld.STT(language) / inworld.TTS(voice, rate, temp, language) / DemoAgent(persona, lang, shots)
→ 开场白:有 persona 用 generate_reply 按人设即兴生成;否则按语言播固定问候
```

### 3.2 单轮对话(注入发生处)

```
用户说话 → Silero VAD 检测 → Inworld STT 流式转写
→ 轮次判定(默认 MultilingualModel 神经网络;小内存环境 TURN_DETECTION=vad 用静音判停)
→ ★ on_user_turn_completed(turn_ctx, new_message):
   1. POST /api/context {text: 用户原话}(300ms 超时,失败降级为无注入)
   2. 命中知识 → 注入 system 消息「回答时可参考以下资料(不相关则忽略):…」
   3. 命中示例 → 注入 system 消息「严格模仿以下示例的语气、风格和长度:示例N 用户/助理:…」
   4. publish_data(topic="context", {enabled, shots, knowledge, chars, items[]}) → 前端观测
→ openai.LLM(gpt-4.1) 流式生成 → inworld.TTS 流式合成 → 播放
→ conversation_item_added → POST /api/utterance 转写入 SQLite
```

注入格式要点:示例**打包为单条 system 消息**而非伪造 user/assistant 轮次——若示例 input 与用户实际提问相同/相近,LLM 会误以为已经回答过而答非所问(实测踩过)。

### 3.3 检索(retrieval.py)

- 模型:`BAAI/bge-small-zh-v1.5`(fastembed/ONNX,免 PyTorch,512 维)
- 索引:启动时全量预计算语料向量,查询时 numpy 点积(余弦相似度);几百条语料暴力计算即微秒级,无需 ANN
- 计分:向量相似度 + 关键词命中加分(每命中 +0.15,最多计 2 个)
- 阈值(按 bge-small-zh 实测校准:同义改写 ~0.47–0.55,无关 <0.42):
  - 风格示例:相似度 ≥0.46 或有关键词命中,取 top 3
  - 知识片段:相似度 ≥0.55,取 top 3(宁多勿漏,注入指令带「不相关则忽略」由 LLM 兜底)
- 降级:检索器初始化失败自动回退纯关键词匹配;单次检索 300ms 超时降级为无注入,检索故障不阻塞对话

### 3.4 音色解析(resolve_voice)

```
voice 输入 ──匹配 ^[A-Za-z0-9_\-.]{1,80}$ ──→ 视为音色名/voice ID,直接透传
        └─其他文字──→ 视为音色描述(仅英文有效):
             POST /voices/v1/voices:design {designPrompt, langCode, previewText, numberOfSamples:1}
             → 取 previewVoices[0].voiceId
             → POST /voices/v1/voices/{id}:publish 转永久(preview 是一次性的)
             → 失败降级默认音色,不阻塞通话
默认音色按语言:zh-CN → Mei(标准普通话女声) / en-US → Ashley
```

## 4. 前端实现

- 技术:Next.js 16(App Router,纯客户端单页)+ Tailwind v4 + `@livekit/components-react`
- 字幕:`useTranscriptions()`(LiveKit 文本流 `lk.transcription` 标准);**说话人归属**用 `streamInfo.attributes["lk.transcribed_track_id"]` 对比本地麦克风 trackSid——用户和 AI 的转写都由 agent 发布,不能用 sender 判断
- 时间戳:本地时钟记每条转写首次出现(start)与最后内容变化(lastChange);AI 转写靠 `lk.transcription_final` 属性封口,用户转写该属性不更新(stream 打开时定格),兜底规则=有更新条目出现即视为已结束
- 延迟徽章:遍历转写序列,记录最近用户条目的 lastChange,下一条 AI 条目的 start 与之相减;只标每轮回答第一段
- 注入观测:`useDataChannel("context")` 收 agent 推送(必须在 `<LiveKitRoom>` 内,经 props 提升到顶层再分发给右栏)
- 助理状态:`useVoiceAssistant().state` → 聆听中/思考中/说话中状态点
- 规范:图标全部内联 SVG(无 emoji/ASCII 图标),大标题 Light 字重

## 5. API 设计

| 端点 | 方法 | 入参 | 出参 | 说明 |
|---|---|---|---|---|
| `/api/token` | POST | room, identity, voice(≤10000), persona(≤10000), rate(0.5–2), temp(0–2), lang(zh-CN\|en-US), shots(bool) | {token, url} | 主入口;含音色解析与配置存储 |
| `/api/token` | GET | room, identity | {token, url} | 脚本/调试用,默认设置 |
| `/api/session-config` | GET | room | 完整设置 JSON | agent 入房后调用 |
| `/api/context` | POST | {text} | {shots[], knowledge[]}(带 score) | RAG 检索 |
| `/api/shots` | POST | {text} | {shots[]} | 兼容旧探针 |
| `/api/voices` | GET | lang(zh\|en) | {voices:[{id,desc}]} | Inworld 音色列表代理,内存缓存 |
| `/api/utterance` | POST | {room, role, content} | {ok, ts} | 转写入 SQLite(带服务器时间戳) |
| `/health` | GET | — | {ok, shots_loaded} | 健康检查 |

数据文件:`few_shots.json`(风格示例:keywords/input/output,现有客服腔×3 + 极简风×3)、`knowledge.json`(知识片段:id/text,现有 FF 车型/公司知识 6 条)。修改后重启 API 自动重建索引。

## 6. 部署方案

### 6.1 形态

| 组件 | 平台 | 部署方式 | 升级方式 | 耗时 |
|---|---|---|---|---|
| frontend | Vercel(项目 livekit-inworld-demo) | CLI | `cd frontend && vercel --prod` | ~30s |
| api + agent | Railway(项目 profound-surprise,单容器) | GitHub 连仓库 | `git push` 自动构建 | ~10min |

### 6.2 容器设计(Dockerfile + start.sh)

- 基础镜像 python:3.12-slim;**模型烘进镜像**(turn-detector + bge-small-zh ONNX),冷启动不下载
- 单容器双进程:start.sh 先起 uvicorn,健康检查通过后起 `agent/main.py start`;任一进程退出容器退出,平台按 Restart Policy 重启
- SQLite 挂 Railway Volume(`/data`)持久化

### 6.3 环境变量(Railway 10 个)

```
LIVEKIT_URL / LIVEKIT_API_KEY / LIVEKIT_API_SECRET   LiveKit Cloud 凭据
INWORLD_API_KEY / OPENAI_API_KEY                      模型服务凭据
BACKEND_URL=http://localhost:8000                     agent → API(容器内)
DB_PATH=/data/utterances.db                           SQLite 持久卷
ALLOWED_ORIGINS=https://livekit-inworld-demo.vercel.app,…   CORS 白名单
PORT=8000        ★必须显式设置(Railway 注入随机 PORT 会致 502)
TURN_DETECTION=vad   ★仅小内存(<2GB)环境;升配后删除恢复神经网络轮次模型
```

Vercel 1 个:`NEXT_PUBLIC_BACKEND_URL=<Railway 域名>`。

### 6.4 资源与成本

- 镜像 ~2GB;运行内存:vad 模式 <1GB,完整模式(神经网络轮次)需 ≥2GB
- Railway Trial(1GB/$5)可跑 vad 模式;Hobby($5/月,8GB)可跑完整模式
- 模型调用成本在 Inworld(STT/TTS/Voice Design)与 OpenAI(gpt-4.1)账户侧

## 7. 实测指标

| 指标 | 实测值 | 验收线 |
|---|---|---|
| RAG 检索延迟(API 内) | ~3.5ms/查询 | <50ms |
| 注入风格切换效果 | 同 agent 三组输入三种风格(极简「多云,26 度。」/客服腔问订单号/默认闲聊) | 风格差异明显 ✅ |
| 语义检索召回 | 「把钱还给我」命中退款示例(关键词匹配无法命中) | — |
| 端到端响应(用户说完→AI 开口) | 常规 2–4s;无意义音频(噪音)轮次 4–6s | 字幕徽章实时可见 |
| Agent 冷启动(首通) | 本地 dev ~20s;Railway 常驻后秒级接单 | <2s 仅热启动可达 |
| token 体积 | 312 字节(配置不入 token) | — |

验证工具:`scripts/talk_probe.py`(macOS `say` 合成中文当麦克风 → 能量阈值判断开场白播完 → 提问 → 校验回复音频与 SQLite 转写,全程无需浏览器)、`scripts/probe_client.py`(TTS 单向冒烟)。

## 8. 已知限制

1. **会话配置表在 API 进程内存**:多实例部署不共享(当前单实例无影响);重启丢失未开始的会话配置
2. **音色描述仅支持英文**(Inworld Voice Design 的 prompt 限制);每次描述生成在 Inworld workspace 留一个永久 voice,需定期清理
3. **vad 轮次模式**(线上当前态)打断与轮次判定靠静音时长,比神经网络模型迟钝;升内存后可恢复
4. **检索阈值与 embedding 模型绑定**:换模型必须重新校准(当前值:shot 0.46/knowledge 0.55,基于 bge-small-zh 实测分布)
5. 噪音/短促音频(<2s)可能被 STT 误识别并以边缘分数误触发示例注入(0.46–0.53 区间);可观测、可通过提高阈值收紧(代价是同义改写漏召回)
6. 转写持久化不保证顺序与幂等(demo 范围,入库带时间戳供事后排序)
